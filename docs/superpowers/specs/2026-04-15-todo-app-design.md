# Todo App — Design Spec
**Date:** 2026-04-15
**Status:** Approved

---

## Overview

A personal todo list web app that runs locally. Data is stored in a `todos.json` file managed by a small Express backend. The frontend is built with React + Vite. Both are started with a single command.

---

## Architecture

**Monorepo structure:**

```
todo-app/
├── frontend/          # React + Vite (runs on http://localhost:5173)
│   └── src/
│       ├── components/
│       │   ├── TodoList.jsx     # Renders list of todo cards
│       │   ├── TodoItem.jsx     # Single todo card with expand/edit/delete
│       │   ├── TodoForm.jsx     # Add/edit modal form
│       │   └── FilterSort.jsx   # Filter and sort controls
│       └── App.jsx              # Root component, manages state and API calls
├── backend/           # Node.js + Express (runs on http://localhost:3001)
│   ├── server.js      # All API routes
│   └── todos.json     # Data file (auto-created on first run if missing)
└── package.json       # Root — starts both with concurrently
```

## Data Model

Each todo item is stored as a JSON object:

```json
{
  "id": "uuid-v4",
  "title": "Buy groceries",
  "notes": "Don't forget milk",
  "priority": "high",
  "status": "todo",
  "dueDate": "2026-04-20",
  "tags": ["personal", "shopping"],
  "createdAt": "2026-04-15T10:00:00Z"
}
```

**Field rules:**
- `id` — UUID v4, generated by backend on creation
- `title` — required, non-empty string
- `notes` — optional string, can be empty
- `priority` — one of: `"high"`, `"medium"`, `"low"`
- `status` — one of: `"todo"`, `"in-progress"`, `"done"`
- `dueDate` — optional, ISO date string `YYYY-MM-DD`
- `tags` — array of lowercase strings, can be empty
- `createdAt` — ISO timestamp, set by backend on creation

## API

Base URL: `http://localhost:3001/api`

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/todos` | Return all todos |
| POST | `/todos` | Create a new todo |
| PUT | `/todos/:id` | Update a todo (any fields) |
| DELETE | `/todos/:id` | Delete a todo |

All requests and responses use JSON. The backend reads and writes directly to `todos.json` on every operation.

## Frontend UI

### Layout

**Top bar:**
- App title: "My Todos"
- "Add Todo" button — opens the add form modal

**Filter & Sort bar:**
- Filter by status: All / Todo / In Progress / Done
- Filter by priority: All / High / Medium / Low
- Filter by tag: dropdown populated from all existing tags
- Sort by: Due Date (ascending) or Priority (high → low)

**Todo list:**
- Cards displayed in sorted order based on current sort selection
- Filtered results update immediately on filter change

### Todo Card

Each card displays:
- Left border color — red (high), yellow (medium), gray (low priority)
- Title
- Priority badge
- Due date (or "No due date")
- Tags as small chips
- Status dropdown — change status inline without opening the edit form
- Edit button — opens the edit form modal pre-filled
- Delete button — removes immediately (no confirmation dialog, it's personal use)

Clicking anywhere else on the card expands it to reveal the notes field.

### Add/Edit Modal Form

Fields:
- Title (text input, required)
- Notes (textarea, optional)
- Due date (date picker, optional)
- Priority (dropdown: High / Medium / Low, default: Medium)
- Tags (text input, comma-separated, e.g. "work, urgent")
- Status (dropdown: Todo / In Progress / Done, default: Todo)

Submit button: "Add Todo" or "Save Changes" depending on mode.

## Error Handling

- If the backend is unreachable on app load, show a top banner: "Cannot connect to server. Make sure the backend is running."
- If a create/update/delete request fails, show an inline error message. Keep the form open on failure so the user doesn't lose their input.
- `todos.json` is auto-created as an empty array `[]` on first run if the file doesn't exist.

## Dev Setup

```bash
# From todo-app/
npm install        # installs root deps (concurrently)
npm run dev        # starts both frontend and backend
```

Vite is configured to proxy `/api` requests to `http://localhost:3001` — no CORS configuration needed.

---

## Out of Scope

- User authentication (personal local tool only)
- Multiple lists or boards
- Drag-and-drop reordering
- Reminders or notifications
- Mobile responsiveness (desktop only for now)
- Unit or integration tests
